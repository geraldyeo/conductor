### TC-07-A: Send to running session ‚Äî PARTIAL

```
$ ao status --json | jq '.[] | select(.id=="ao-7f89307b4ba0a57") | .status'
"working"

$ ao send ao-7f89307b4ba0a57 "Please summarize what you have done so far."
error: orchestrator returned error: [internal_error] runtime error: command failed: can't find pane: ao-7f89307b4ba0a57
EXIT:1
```

**Root cause (MVP limitation, not a new bug):** `claude -p` (inline mode) exits within seconds, destroying the tmux pane. By the time `ao send` executes the `tmux send-keys` call, the pane no longer exists. The IPC path (finding the session as `working`) works correctly; the tmux delivery is a timing race inherent to single-turn inline mode.

`ao send` is designed for the post-launch/interactive mode (post-MVP). In inline mode, the viable send path is TC-07-E (direct tmux fallback without orchestrator), which already passes.

TC-07-B (send from file) and TC-07-C (--no-wait) have the same root cause and were not re-run separately.

### TC-10-A/B: Session cleanup ‚Äî **FAIL** (new Bug-7)

```
$ ao spawn 12
Session spawned: ao-7f89407b4ba0c0a  [status: working]

$ gh issue close 12
‚úì Closed issue geraldyeo/conductor#12

$ ao session cleanup --dry-run
Killed:
Skipped: ao-7f89407b4ba0c0a
EXIT:0
```

Session was `working`, tracker issue was `CLOSED` ‚Äî expected to appear in "Killed" list. Instead: skipped.

**Root cause (Bug-7):** `TrackerConfig.terminal_states` has `#[serde(default)]` (empty Vec). The `ao init` generated config does not include a `terminalStates` field. At runtime `p.terminal_states` is `[]`. `classify_state("CLOSED", &[])` always returns `Active` ‚Äî no session is ever classified as cleanup-eligible. Same empty Vec means the `Done` state is also unreachable via the poll loop.

Note: the poll DID correctly transition `ao-7f89407b4ba0c0a` to `terminated` ‚Äî but via the `Exited` edge (agent process ended), not via the `Terminal` tracker-state check. So session lifecycle still progresses, just through the wrong terminal state.

See Bug-7 below.

## New Bug Found

### Bug-7 ‚Äî HIGH: `ao init` generated config omits `terminalStates`, making cleanup and `Done` state unreachable

**File:** `packages/core/src/config/schema.rs:199` and init config template

`TrackerConfig.terminal_states` is `#[serde(default)]` ‚Äî empty Vec when the field is absent from YAML. The generated config from `ao init` does not include `terminalStates: [closed]` under `tracker:`. As a result:

1. `ao session cleanup` always classifies all sessions as `Active` ‚Üí skips every session
2. `classify_state` in the poll loop also uses `p.terminal_states`, so `TrackerState::Terminal` is never reached via polling
3. The `Working ‚Üí Done` edge (`Exited && Terminal`) is unreachable; sessions exit to `Terminated` regardless of whether the issue was closed
4. TC-11 full lifecycle (`pr_open ‚Üí done`) is blocked

**Fix options:**
- Change `#[serde(default)]` on `terminal_states` to a function default: `fn default_terminal_states() -> Vec<String> { vec!["closed".to_string()] }`
- Or: include `terminalStates: [closed]` in the init config template

**Impact:** TC-10-A/B FAIL, TC-11 PARTIAL, `Done` state unreachable in all default deployments.

### TC-07-A: Send to running session â€” PARTIAL

```
$ ao status --json | jq '.[] | select(.id=="ao-7f89307b4ba0a57") | .status'
"working"

$ ao send ao-7f89307b4ba0a57 "Please summarize what you have done so far."
error: orchestrator returned error: [internal_error] runtime error: command failed: can't find pane: ao-7f89307b4ba0a57
EXIT:1
```

**Root cause (MVP limitation, not a new bug):** `claude -p` (inline mode) exits within seconds, destroying the tmux pane. By the time `ao send` executes the `tmux send-keys` call, the pane no longer exists. The IPC path (finding the session as `working`) works correctly; the tmux delivery is a timing race inherent to single-turn inline mode.

`ao send` is designed for the post-launch/interactive mode (post-MVP). In inline mode, the viable send path is TC-07-E (direct tmux fallback without orchestrator), which already passes.

TC-07-B (send from file) and TC-07-C (--no-wait) have the same root cause and were not re-run separately.

## Bug Found and Fixed

### Bug-7 â€” HIGH: `ao init` generated config omits `terminalStates` â€” **FIXED**

**File:** `packages/core/src/config/schema.rs`

**Root cause:** `TrackerConfig.terminal_states` had `#[serde(default)]` â€” empty Vec when the field is absent from YAML. `classify_state("CLOSED", &[])` always returned `Active`.

**Fix:** Added `fn default_terminal_states() -> Vec<String> { vec!["closed".to_string()] }` and changed the field attribute to `#[serde(default = "default_terminal_states")]`. Config without an explicit `terminalStates` key now correctly defaults to `["closed"]`.

**Test:** `test_tracker_default_terminal_states_includes_closed` written first (RED), confirmed passing after fix (GREEN). All 85 tests pass.

**Verified:** TC-10-A dry-run correctly lists working session (closed issue) in Killed. TC-10-B kills the session. `Done` state and cleanup now reachable in all default deployments.

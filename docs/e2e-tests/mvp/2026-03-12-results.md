## Bugs Found

### Bug-1 — CRITICAL: `Working → Killed` fires before `Working → Terminated`/`Done` on natural exit

**File:** `packages/core/src/lifecycle/graph.rs:99`

**Root cause:** Edge ordering in `Working` node:
- Edge 7: `Working → Killed` guard `!runtime_alive` (precedence 7)
- Edge 8: `Working → Done` guard `activity_state == Exited && tracker_state == Terminal` (precedence 8)
- Edge 9: `Working → Terminated` guard `activity_state == Exited` (precedence 9)

When `runtime_alive = false`, the gather loop sets `activity_state = Exited`. Both the edge-7 guard (`!runtime_alive`) and the edge-8/9 guards (`Exited`) are simultaneously true. Edge 7 wins because its precedence number is lower. Result: every naturally-completed session is marked `killed` instead of `terminated` or `done`. The `Terminated` and `Done` states are unreachable from `Working`.

**Fix:** Reorder so `Done` → `Terminated` → `Killed` within the `Working` node:
```
edge 7: Working → Done       guard: Exited && Terminal  (was prec 8)
edge 8: Working → Terminated guard: Exited              (was prec 9)
edge 9: Working → Killed     guard: !runtime_alive      (was prec 7)
```

**Impact:** TC-05-A partial, TC-07-A/B/C untestable, TC-10 untestable, TC-11 blocked.

### Bug-2 — HIGH: `ao init` interactive mode unimplemented

**File:** `packages/cli/src/commands/init.rs`

`ao init` without `--auto` immediately errors with "Interactive mode not yet implemented. Use --auto flag." The `-o` flag is also broken for the same reason (it checks `--auto` before respecting `-o`).

### Bug-3 — HIGH: `ao status -p <unknown>` returns exit 0

**File:** `packages/cli/src/commands/status.rs`

`status` does not call `resolve_project()`. When `-p nonexistent` is specified, it silently iterates over zero matching projects and prints "No sessions." Expected: error "project not found", exit 3.

**Impact:** TC-12 exit code for "unknown project ID" is 0 instead of 3.

### Bug-4 — HIGH: `ao session kill` does not check terminal state

**File:** `packages/core/src/orchestrator/mod.rs:483`

`handle_kill` unconditionally overwrites the session status to `Killed` and calls `runtime.destroy()`. No guard on whether the session is already in a terminal state. Killing an already-killed session returns "Kill scheduled", exit 0.

**Impact:** TC-09-B FAIL.

---

### Bug-5 — HIGH: `ao spawn` attach hint shows wrong tmux session name

**File:** `packages/cli/src/commands/spawn.rs:52`

Spawn output prints `Attach: tmux attach -t {session_id}` but the actual tmux session name is `ao-{session_id}` (the `TmuxRuntime::tmux_session_name()` function prepends `ao-`, creating a double prefix since session IDs already start with `ao-`).

Confirmed via error message when sending to a killed session: `can't find pane: ao-ao-243b720719617b18`.

**Fix options:**
1. Strip the `ao-` prefix from session IDs so the tmux name is just `ao-{hash}`, or
2. Update the attach hint to use `tmux_session_name(session_id)` instead of bare `session_id`.

**Impact:** Misleading UX; `tmux attach` from the hint silently connects to the wrong session.

---

### Bug-6 — MINOR: Worktree destroy does not delete the git branch

**File:** `packages/core/src/plugins/workspace/worktree.rs`

When a session is killed/cleaned up, the git worktree directory is removed but the branch is not deleted. Subsequent attempts to spawn the same issue (after manual session cleanup) fail with `git: fatal: a branch named '...' already exists`. Requires manual `git worktree remove` + `git branch -d` to fully clean up.

## Observations

- Session IDs are deterministic (FNV-1a hash of issue URL), so the same issue URL always maps to the same session ID. Once a session directory exists (even in a terminal state), re-spawning the same issue is blocked until the session is archived/deleted. This is intentional by design but has UX implications for iterative testing.
- `ao stop` when not running returns exit 0 (idempotent) — correct per spec.
- `ao status` reads SessionStore directly without requiring the orchestrator — correct per spec.
- IPC concurrent request handling is correct: atomic `create_dir` prevents duplicate sessions under concurrent load.
- Config validation (garde) provides field-level error detail — good DX.

# ADR-0007: CLI

## Status
Accepted

## Context

The CLI (`ao`) is the user-facing surface that ties together all prior subsystems: the session lifecycle engine (ADR-0001), configuration system (ADR-0003), plugin system (ADR-0004), workspace and session metadata (ADR-0005), and tracker integration (ADR-0006). It serves two audiences: humans interacting via terminal, and scripts/supervisor agents consuming structured JSON output.

Six prior ADRs constrain the design:

1. **ADR-0001 (Session Lifecycle Engine)** defines the poll loop (30s default, 3-phase: gather/evaluate/transition), `PollContext.manualKill` (global edge at precedence 0), and crash recovery via re-poll on restart.
2. **ADR-0002 (Implementation Language)** locks in Rust, `clap` for CLI parsing, `tokio` for async, and `CommandRunner` for subprocess management.
3. **ADR-0003 (Configuration System)** defines `load()`, `discover_config_path(start_dir)`, `generate_default()`, and walk-up-tree config discovery.
4. **ADR-0004 (Plugin System)** defines static factory functions (`create_agent()`, `create_runtime()`), `LaunchPlan` execution, `RuntimeStep::SendMessage`, `Agent::detect_activity()`, and `Runtime::execute_step()`.
5. **ADR-0005 (Workspace & Session Metadata)** defines `SessionStore` (CRUD + list + archive), `DataPaths` (path computation + `ensure_dirs()`), and the 10-step session creation sequence.
6. **ADR-0006 (Tracker Integration)** defines `create_tracker()` with fail-fast validation, `get_issue()` + `classify_state()` for pre-spawn validation, `branch_name()` for branch derivation, and `add_comment()` for workpad comments.

Key forces:

- The PRD (FR6) specifies 13 commands. Not all are MVP — `ao dashboard`, `ao open`, `ao review-check`, and `ao session restore` depend on undelivered FRs.
- `ao start` creates the orchestrator process. The PRD specifies an orchestrator-as-session (FR13) — an AI agent that coordinates other agents — but this depends on ADR-0008 (Prompt System) which hasn't landed. Without FR5 (Scheduling), the orchestrator agent has nothing to autonomously dispatch.
- `ao send` requires busy detection and delivery verification (PRD). This involves reading agent activity state and coordinating with the runtime — the same logic the poll loop uses. Two independent processes (CLI and orchestrator) accessing the same tmux session risks race conditions.
- Multiple projects in a single config require a project resolution strategy that's ergonomic for single-project setups (the common case) without ambiguity for multi-project setups.

## Considered Options

### Process Model (`ao start`)

1. **Foreground process** — `ao start` runs the poll loop in the foreground (blocking). Ctrl-C or SIGTERM stops it. Simple, composable, easy to debug.

2. **Self-daemonizing** — `ao start` forks, writes a PID file, detaches from the terminal. More "service-like" but daemonizing in Rust is non-trivial, and macOS `launchd` discourages it.

3. **Hybrid** — Foreground by default, `--daemon` for background mode. Two code paths to maintain.

### Project Resolution

4. **Explicit `-p` flag always** — Every project-scoped command requires `-p <project>`. Unambiguous but verbose for single-project setups.

5. **Auto-resolve with override** — Single project → use it. CWD inside a project's path → use that project. Multiple ambiguous → error. `-p` always overrides.

6. **Default project in config** — A `defaultProject` config field. Adds a config field only relevant to CLI ergonomics.

### `ao send` Delivery

7. **CLI-side polling loop** — `ao send` independently constructs Agent + Runtime instances, polls activity state, and delivers. Self-contained but risks race conditions with the orchestrator's poll loop — two processes accessing the same tmux session.

8. **Delegate to orchestrator via IPC** — `ao send` sends a request to the running orchestrator, which owns Agent + Runtime instances and coordinates delivery. Serializes access, no races, cached state. Falls back to direct delivery when orchestrator isn't running.

9. **Direct runtime call, no activity check** — `ao send` calls `Runtime::execute_step(SendMessage)` directly without checking activity state. Simplest but loses busy detection.

## Decision

**Process model:** Option 1 — foreground. `ao start` blocks, logs to stdout. At MVP scale, users run it in a terminal tab. If daemon mode is needed post-MVP, `--daemon` is an additive flag.

**Project resolution:** Option 5 — auto-resolve with override. Single-project configs just work. Multi-project configs get CWD-based auto-detection with `-p` as escape hatch.

**`ao send` delivery:** Option 8 — delegate to orchestrator via IPC, with direct fallback. This establishes the orchestrator as the single coordinator — the architectural foundation for the end-state where Conductor runs autonomously (supervisor agent spawns `ao start`, walks away).

**The design has five components:**

### 1. IPC Control Plane

The orchestrator listens on a Unix domain socket at `~/.agent-orchestrator/orchestrator.sock` (orchestrator-wide, not per-project — `DataPaths` is per-project but the orchestrator is a single process managing all projects). Length-prefixed JSON, request-response per connection. No persistent connections.

```rust
#[derive(Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum OrchestratorRequest {
    Spawn { project_id: String, issue_id: String, agent: Option<String>, open: bool },
    BatchSpawn { project_id: String, issue_ids: Vec<String>, agent: Option<String>, open: bool },
    Send { session_id: String, content: String, no_wait: bool, timeout_secs: u64 },
    Kill { session_id: String },
    Cleanup { project_id: String, dry_run: bool },
    Stop,
}

#[derive(Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum OrchestratorResponse {
    Ok { message: String },
    SpawnResult { session_id: String, branch: String, workspace_path: String },
    BatchSpawnResult { results: Vec<BatchSpawnItem> },
    CleanupResult { killed: Vec<String>, skipped: Vec<String> },
    SendResult { delivered: bool, activity_state: String },
    Error { code: String, message: String },
}

#[derive(Serialize, Deserialize)]
pub struct BatchSpawnItem {
    pub issue_id: String,
    pub outcome: BatchSpawnOutcome,
}

#[derive(Serialize, Deserialize)]
#[serde(tag = "result", rename_all = "snake_case")]
pub enum BatchSpawnOutcome {
    Spawned { session_id: String, branch: String },
    Skipped { reason: String },
    Failed { error: String },
}
```

Why Unix domain socket over HTTP: lower overhead, no port conflicts, file-permission-based access control, socket file doubles as liveness check. Why not gRPC: heavy dependency for local-only IPC; JSON over UDS is ~50 lines of code.

**Socket lifecycle:** `ao start` creates and listens. `ao stop` connects, sends `Stop`, orchestrator drains the current poll tick, removes socket file, exits. Stale socket detection: if socket file exists but connection refused, the CLI deletes the stale file and reports "orchestrator not running."

**Command routing:**

| Command | IPC? | Rationale |
|---------|------|-----------|
| `ao init` | No | Config generation, no orchestrator needed |
| `ao start` | No (is the server) | Creates the socket |
| `ao stop` | Yes | Sends `Stop` request |
| `ao status` | No | Reads SessionStore files directly |
| `ao spawn` | Yes | Orchestrator owns session creation sequence |
| `ao batch-spawn` | Yes | Delegates to orchestrator's spawn logic |
| `ao send` | Yes (fallback: direct) | Orchestrator owns activity detection + delivery |
| `ao session ls` | No | Reads SessionStore files directly |
| `ao session kill` | Yes | Sets `manualKill` in PollContext |
| `ao session cleanup` | Yes | Needs tracker + kill coordination |

Read-only commands (`ao status`, `ao session ls`) work without the orchestrator — they read `SessionStore` files directly (loading config via `config::load()`, constructing `DataPaths` per project, and iterating `SessionStore::list()` — lightweight, no plugin construction). Mutating commands require the orchestrator, except `ao send --no-wait` which falls back to a direct `tmux send-keys` call via `CommandRunner` (no `Runtime` construction — tmux is the only MVP runtime, and `send-keys` only needs the session name, which is read from `SessionMetadata`). A warning is printed: "Orchestrator not running, delivering without busy detection."

### 2. Orchestrator Struct

The orchestrator loop lives in `packages/core/src/orchestrator/`, not in `cli`. The `ao start` command constructs it and calls `run()`.

```rust
pub struct Orchestrator {
    config: Arc<Config>,
    stores: HashMap<String, SessionStore>,
    plugins: HashMap<String, ProjectPlugins>,  // tracker + agent + runtime + workspace per project
    socket_path: PathBuf,
    /// IPC requests are sent here by the IPC listener and processed by the poll loop task.
    ipc_tx: mpsc::Sender<(OrchestratorRequest, oneshot::Sender<OrchestratorResponse>)>,
    ipc_rx: mpsc::Receiver<(OrchestratorRequest, oneshot::Sender<OrchestratorResponse>)>,
}
```

**Concurrency model — channel-based serialization.** The `run()` method spawns two concurrent tasks: the IPC listener and the poll loop. Both need access to session state (`stores`, `plugins`, `PollContext` flags like `manualKill`). Rather than locking shared state, the IPC listener sends requests via an `mpsc` channel to the poll loop task, which processes them between ticks. This serializes all mutations without locks:

```
IPC listener task                     Poll loop task
─────────────────                     ──────────────
accept connection ──→ mpsc::send ──→  drain channel (between ticks)
wait on oneshot   ←── oneshot::send ←── handle request, mutate state
send response                         continue to next tick
```

The poll loop drains pending IPC requests at the start of each tick (before the gather phase). Each request includes a `oneshot::Sender` for the response. This guarantees: (a) no concurrent mutation of session state, (b) no lock ordering issues, (c) IPC response latency is bounded by one poll interval + request handling time.

The `shutdown` watch channel unifies Ctrl-C (via `tokio::signal::ctrl_c()`) and `ao stop` (via IPC `Stop` request) into a single shutdown trigger. SIGHUP is reserved for post-MVP config hot-reload (ADR-0003 deferral) and is currently ignored. SIGTERM is handled via `tokio::signal::unix::signal(SignalKind::terminate())`.

`ao start` startup sequence:

1. `config::load()` — discover + load + validate (ADR-0003)
2. Validate plugins — `create_tracker()`, `create_runtime()`, `create_workspace()` per project. Fail-fast.
3. `DataPaths::ensure_dirs()` per project — collision detection via `.origin` (ADR-0005)
4. Create `SessionStore` per project
5. Bind socket — error if already bound (another instance running)
6. Load non-terminal sessions — crash recovery (ADR-0001)
7. Run one immediate poll tick
8. Start poll loop + IPC listener
9. Print status

Graceful shutdown: stop accepting connections, drain current tick, remove socket, exit 0.

### 3. Command Structure

`clap` v4 with derive macros. Global flags on the root command:

| Flag | Type | Default | Purpose |
|------|------|---------|---------|
| `--config <path>` | `Option<PathBuf>` | `None` | Override config discovery |
| `--project <id>` / `-p` | `Option<String>` | `None` | Override project auto-resolution |
| `--json` | `bool` | `false` | JSON output for all commands |
| `--verbose` / `-v` | `bool` | `false` | Debug-level logging to stderr |

`--json` is global, not per-command. The PRD mentions it only on `ao status`, but scripts and supervisor agents benefit from JSON on every command. With `--json`, stdout is always valid JSON; logs/diagnostics go to stderr.

MVP commands (10):

- `ao init [--auto] [--smart] [-o <path>]` — generate config via `generate_default()`
- `ao start [--no-dashboard]` — poll loop + IPC listener (no orchestrator-as-session at MVP)
- `ao stop` — send `Stop` via IPC
- `ao status [-p <project>]` — session table from SessionStore
- `ao spawn <issue> [--agent <name>] [--open]` — spawn session via IPC
- `ao batch-spawn <issues...> [--agent <name>] [--open]` — batch spawn with dedup, 500ms delay between spawns
- `ao send <session> [message...] [-f <file>] [--no-wait] [--timeout <secs>]` — message delivery via IPC
- `ao session ls [-p <project>]` — alias for `ao status`
- `ao session kill <session>` — set `manualKill` via IPC
- `ao session cleanup [-p <project>] [--dry-run]` — kill terminal-tracker sessions via IPC

**`ao send` delivery flow** (orchestrator-side):

1. Look up session's Agent + Runtime instances
2. Poll `detect_activity()` until idle (or timeout, default 600s)
3. Clear partial input: `Runtime::execute_step(SendMessage { content: "\x15" })` (Ctrl-U)
4. Deliver: `SendMessage` for short content, `SendBuffer` for large (>1KB)
5. Verify: 3 retries checking activity state transitions to `active`
6. Return `SendResult { delivered, activity_state }`

**`ao session kill` is not immediate.** It sets `manualKill = true` in PollContext. The next poll tick evaluates the global edge at precedence 0 → transitions to `killed` → entry action destroys runtime + workspace. This avoids duplicating cleanup logic. After setting the flag, the CLI briefly polls `SessionStore` (up to 5s, checking every 500ms) for the session to reach `killed` status. If confirmed within 5s, it prints "Session killed." If not, it prints "Kill scheduled, will complete within {poll_interval}s." This provides immediate feedback for the common case (poll tick runs within seconds) without blocking indefinitely.

**`ao session cleanup` flow** (orchestrator-side):

1. `SessionStore::list()` → filter non-terminal sessions
2. For each: `tracker.get_issue()` + `classify_state()`
3. If terminal → set `manualKill = true`
4. Return killed/skipped lists

### 4. Error Handling & Exit Codes

| Code | Meaning | Examples |
|------|---------|---------|
| 0 | Success | Command completed, `ao stop` when already stopped |
| 1 | General error | Spawn failed, send timeout, IPC error |
| 2 | Usage error | Invalid args, missing required arg (clap default) |
| 3 | Config error | Config not found, validation failed, unknown project |
| 4 | Orchestrator not running | Mutating command sent but no orchestrator |

Human mode errors: `error:` prefix, one-line summary, indented `→` context lines. JSON mode errors: structured JSON on stdout with `code` and `message` fields.

Output conventions: stdout for results (tables or JSON), stderr for `error:`/`warning:`/debug logs. Progress indicators to stderr only, suppressed in JSON mode.

Dependency direction: `cli` depends on `core`. Mutating operations go through IPC to the `Orchestrator` (in `core`), not by calling core internals directly. Read-only operations access `SessionStore` directly.

## Consequences

Positive:

- The IPC control plane establishes the orchestrator as the single coordinator — all mutating operations go through it, serializing access to sessions and runtimes. This eliminates race conditions between independent CLI invocations and positions the architecture for the end-state where a supervisor agent (or orchestrator-as-session) drives Conductor autonomously.
- Read-only commands (`ao status`, `ao session ls`) work without the orchestrator running. Users can inspect state even if the orchestrator crashed. This separation (read-only = direct file access, mutating = IPC) is clean and allows progressive complexity.
- The Unix domain socket is lightweight infrastructure that serves multiple purposes: IPC transport, liveness detection (socket file exists = orchestrator running), and access control (file permissions). No PID file, no port allocation, no HTTP server.
- Project auto-resolution makes single-project configs zero-friction (no `-p` flag needed) while remaining unambiguous for multi-project setups. CWD-based detection leverages the `path` field already validated in ADR-0003.
- `--json` as a global flag means every command is scriptable. Supervisor agents and CI pipelines can parse `ao spawn --json` output without screen-scraping tables. The stdout/stderr split (results vs. logs) is a standard Unix convention.
- `ao session cleanup` with `--dry-run` provides a safe audit tool. Users can see what would be killed before committing. The implementation is trivial — it reuses `SessionStore::list()`, `get_issue()`, and `classify_state()` from existing ADRs.
- The `Orchestrator` struct in `core` (not `cli`) keeps the poll loop, IPC handling, and session coordination testable without the CLI layer. `ao start` is a thin wrapper that constructs and runs it.
- The channel-based concurrency model (IPC listener → mpsc → poll loop) serializes all mutations without locks. No `Mutex`, no `RwLock`, no lock ordering. IPC requests are processed between poll ticks, guaranteeing no concurrent access to session state.
- The foreground process model is the simplest to implement, debug, and compose. It follows the convention of `cargo watch`, `next dev`, and `tilt up`. Daemon mode is an additive post-MVP flag.

Negative:

- Mutating commands fail when the orchestrator isn't running (exit code 4). Users must run `ao start` before `ao spawn`. Mitigated by a clear error message: "Orchestrator not running. Start it with `ao start`." The `ao send --no-wait` fallback provides a limited escape hatch.
- `ao session kill` is not immediate — it sets a flag and waits up to 30s for the next poll tick. Users expecting instant kill may be surprised. Mitigated by: (a) the CLI prints "Kill requested, will take effect within 30s" and (b) the alternative (immediate kill in the CLI process) would duplicate cleanup logic from the lifecycle engine.
- The IPC protocol is a new surface area to maintain and version. Adding a new command requires updating both the request/response enums and the handler. At MVP scale (10 commands) this is manageable. Post-MVP, a version field on requests would allow backward-compatible evolution.
- `ao status` reads `SessionStore` files directly, bypassing the orchestrator. If the orchestrator has uncommitted state (e.g., mid-transition), `ao status` may show stale data. The staleness window is at most one poll tick (30s) — the same convergence trade-off accepted in ADR-0001.
- Project auto-resolution via CWD matching depends on `ProjectConfig.path` being an absolute path that matches the user's actual CWD after canonicalization. Symlinked directories or unusual mount points may cause mismatches. `-p` is the reliable override.
- The `ao init --smart` flag is a no-op at MVP. Users who discover it via `--help` may be disappointed. Mitigated by printing "Smart detection coming in a future release; using auto mode."
- The foreground process model means `ao start` occupies a terminal. Users must manage backgrounding themselves (tmux tab, `&`, etc.). This is acceptable for a developer tool but less polished than a self-daemonizing service.
- `ao send` fallback mode (without orchestrator) skips busy detection entirely, falling back to a direct `tmux send-keys` call. A message delivered while the agent is mid-thought may cause confusion. The warning makes this explicit, and the fallback only activates when the orchestrator is not running — users won't hit it accidentally during normal operation.
- IPC request latency is bounded by the poll interval: a `Spawn` request sent just after a tick starts must wait for the tick to complete before being processed. At 30s intervals, worst-case latency is ~30s for spawn. Mitigated by keeping tick execution fast (gather phase is the bottleneck, bounded by subprocess timeouts). If latency becomes a concern, the poll loop can check the channel mid-tick between sessions.
- The `\x15` (Ctrl-U) clear-input step in `ao send` assumes Unix terminal line discipline (readline). This works for Claude Code + tmux at MVP. Post-MVP, the clear-input step could be made agent-specific via an `Agent::clear_input_step() -> Option<RuntimeStep>` method.
